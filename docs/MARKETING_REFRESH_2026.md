# Marketing architecture and conversion messaging (2026)

This document captures the current landing strategy in a durable format.
It replaces task-oriented sprint logs.

## Positioning in one sentence

Open-source, low-friction cryptographic evidence for software creation time and integrity, verifiable independently from the service provider.

## Priority objections to resolve on-page

1. Trust objection: "What if this service disappears?"
2. Relevance objection: "Is this for my type of work?"
3. Comparison objection: "How is this different from existing options?"
4. Pricing objection: "Is there a hidden paywall?"

## Target section order

1. Hero (with risk-reversal line)
2. Trust bar
3. Deposit flow section
4. How it works
5. Use cases
6. Legal framing
7. Comparison table
8. Why Bitcoin / independent verification
9. Trust grid
10. Pricing strip
11. History / timeline
12. FAQ
13. Final CTA

## Core content blocks

### Hero risk reversal

Must explicitly state evidence survivability independent of this website.

### Use-cases grid

Maintain 4 personas:

- Indie developer
- Designer/freelancer
- Writer/researcher
- IP/legal team

Each card should include:

- scenario context
- expected artifact type
- immediate call to action

### Comparison table

Maintain direct comparison against typical alternatives:

- notary workflow
- state registration path
- private registry/deposit platforms
- qualified TSA path

Focus rows:

- cost
- issuance time
- verification model
- provider-independence
- legal maturity
- openness/transparency

### Pricing strip

Single-card pricing message:

- clear `$0` primary message
- "free forever" and no-account signal
- premium roadmap framed as future team offering

### Open Graph asset

- Keep dedicated OG image at `frontend/public/images/og-image.png`.
- Preserve readability at compressed preview size.
- Keep branding and value proposition legible in first glance.

## EN/RU language policy for marketing pages

- English is source language for copy decisions.
- Russian translations must preserve legal meaning, not just literal wording.
- Any new landing section must ship with both locale keys in the same release.

## Quality checks before publish

1. Verify visual order and section anchors on desktop and mobile.
2. Verify EN/RU parity for all headings, labels, and CTA text.
3. Validate OG tags and preview image availability.
4. Ensure table remains usable on small screens (horizontal container scroll is acceptable).
